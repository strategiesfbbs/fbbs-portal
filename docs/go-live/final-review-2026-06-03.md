### [1] (MEDIUM) Correctness / Bond swap math — End-of-month coupon dates drift via setUTCMonth overflow, corrupting accrued interest and coupon schedules
- **File:** `server/swap-math.js` — `server/swap-math.js:143`, `server/swap-math.js:231`, `server/swap-math.js:236`, `server/swap-math.js:274`, `server/swap-math.js:277`
- **Impact:** Accrued interest, derived YTW/YTM, and modified duration are wrong for any held or offered bond with a month-end coupon date — a large fraction of munis and agencies pay on the 1st/15th/end-of-month. The error feeds solveBuyParForProceeds (cash-neutral sizing off by hundreds of dollars), the printed proposal economics, and the send-time snapshot.
- **Fix:** Add a month-end-aware step helper near the other date helpers and replace all four raw setUTCMonth walks with it. Add a regression case to tests/swap-math.test.js for a month-end maturity (e.g. inferLastCouponDate('2030-08-31',...) === 2026-02-28 and the corresponding accruedInterest ~8333).
- **Risk:** Low. The helper changes results only for month-end maturities that previously overflowed; all other dates are unaffected (verified mid-month control is identical). One nuance: the proposed helper carries the clamped day forward (e.g. Dec31 walked back lands on Jun30 then keeps 30), so it does not perfectly re-anchor to the original day-of-month per strict EOM convention — but it is strictly closer to correct than the current overflow and removes the dollar error. Any tests/swap-math.test.js golden values that were silently computed against the buggy walk for month-end inputs would need updating, but those values are currently wrong. No other call sites beyond the four in this file.

### [2] (MEDIUM) Correctness / Swap store + render — cancelProposal has no status guard — an executed (settled) trade can be silently re-cancelled
- **File:** `server/server.js` — `server/server.js:5556-5578`, `server/swap-store.js:661-668`

Optionally also defense-in-depth the store: scope cancelProposal's UPDATE to `WHERE id=? AND status IN ('draft','sent')` and check result.changes so the store refuses an out-of-lifecycle transition even if a future caller skips the handler guard.
- **Risk:** Low. The change only rejects transitions that should never have been allowed. No swap-store/swap-math regression tests exercise cancelling an executed/cancelled proposal (cancelProposal is exported but the lifecycle guard lives in the handler), so adding the guard won't break existing tests. The only behavioral change a caller could notice is the new 409/idempotent-200 on out-of-lifecycle cancels, which is the intended correction. If the store UPDATE is also scoped, ensure getProposal is still returned (it is) so the response shape is unchanged.

### [3] (MEDIUM) Correctness / Swap store + render — Render falls back to LIVE legs whenever snapshot is absent — including for executed/cancelled proposals, so a 'frozen' artifact can silently drift
- **File:** `server/swap-render.js` — `server/swap-render.js:339-363`, `server/swap-render.js:369-371`
- **Impact:** The immutability guarantee only holds for the 'sent' window. - **Risk:** Very low. The existing swap-render test only asserts the `sent` case (tests/swap-render.test.js:85-102), which the new condition still satisfies; no test covers executed/cancelled render. Behavioral change is limited to executed/cancelled proposals now rendering from snapshot instead of live — which is the intended invariant. The draft path and the no-snapshot cancelled-from-draft path are unaffected because the `record.snapshot &&` guard remains, so they still hit the live branch with no DRAFT watermark (same as today).

### [4] (MEDIUM) Correctness / PDF parsers — toPercentNumber turns a literal 1.00% CD rate into 100%
- **File:** `server/cd-offers-parser.js` — `server/cd-offers-parser.js:435-442`
- **Impact:** A 1.00% offering is published as 100.000% on the CD Explorer / tear sheets — a wildly wrong rate surfaced to reps and downstream swap math. Silent: no warning is emitted.
